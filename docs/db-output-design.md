# 온톨로지 DB — output-first 설계

> 2026-07-19. "쌓기만 하는 DB"를 막기 위한 설계 원칙과 논리 스키마.
> 배경: 잡스랩 저장소 `docs/ontology/kit.md`(진단 키트 v0.1) · `docs/superpowers/specs/2026-07-19-ontology-server-design.md`

## 1. 현재 상태 진단 (솔직하게)

| 구성요소 | 상태 |
|---|---|
| 인터뷰 수집 (guide → `/api/ontology/interview`) | 작동 — 노드를 **뽑아내는** 쪽은 있음 |
| 저장 | 이 저장소엔 DB 없음. 수집된 노드가 어디에 어떤 형태로 쌓이는지 미정의 |
| 소비 | **0.** mock 대시보드는 하드코딩 데이터. 수집→저장→산출로 이어지는 경로가 없음 |

**진단: 지금 구조는 input에서 끝난다.** 인터뷰가 아무리 잘 돼도, 쌓인 그래프를 읽는 소비자가 없으면 온톨로지는 "정리해둔 문서"와 다르지 않다. 이 문서는 그 반대 — **산출물에서 거꾸로 스키마를 결정**한다.

## 2. 원칙: 질문이 스키마를 결정한다

> **이 DB에 던질 질문 목록을 먼저 고정하고, 그 질문에 답하는 데 필요한 필드만 받는다.**
> 어떤 질문에도 쓰이지 않는 필드는 받지 않는다 — 이게 데이터 층의 Minimum Input이다.

## 3. 산출물 5종 (이 DB가 존재하는 이유)

| # | 산출물 | 던지는 질문 | 소비자 | 돈과의 연결 |
|---|---|---|---|---|
| O1 | **병목 리포트** | "이 회사에서 시간·돈이 어디서 얼마나 새나?" — **시간 해방(환산 가치)과 현금 누수(실질 절감) 두 칸으로 분리 보고** | 진단 고객 (A 상품 산출물 ②) | A 매출의 근거 + 정직성 차별화 |
| O2 | **개선 제안서** | "각 병목에 어떤 처방이 있고, 뭐부터 고치면 회수가 빠른가?" | 고객 + 영업 (스코프 방어 문구의 '다음 계약 목록') | A→C 다리 |
| O3 | **에이전트 컨텍스트 팩** | "이 회사의 업무 구조·규칙·용어를 에이전트가 알려면 뭘 줘야 하나?" | 자동화 에이전트 (jurin, 구축 시스템) | C 구독의 엔진 |
| O4 | **패턴 승격** | "2개사 이상에서 반복된 병목 구조는 무엇인가?" | 잡스랩 자신 (키트 카탈로그 확장) | 진단 원가 하락 = 해자 |
| O5 | **전후 비교** | "개선 전/후로 낭비가 실제로 얼마나 줄었나?" | 고객 + 영업 (사례 공개, 갱신 설득) | 구독 유지 + 공개 사례 |

## 4. 논리 스키마 (기술 중립 — 그래프 DB든 관계형이든 동일 적용)

### nodes
키트 v0.1 그대로 + 산출을 위한 필드 추가.

| 필드 | 타입 | 어느 산출물이 요구하나 |
|---|---|---|
| `tenant` | string, 필수 | 전부 (격리) |
| `id` | string (A-01 형식), `(tenant,id)` 유니크 | 전부 |
| `type` | Actor·Object·Event·Record·Rule·Bottleneck | 전부 |
| `name` / `desc` | string | 전부 |
| `status` | 확인됨 \| 미확인 | O1 (미확인 노드는 리포트에 ❓로 — 정직성) |
| `evidence` | string — **인터뷰 발화 인용** | O1·O2 (근거 없는 병목은 고객이 안 믿음) |
| `created_at` / `updated_at` | timestamp | O5 |

### nodes: Bottleneck 전용 (★ 이 3개 없으면 O1이 불가능)

| 필드 | 예 | 용도 |
|---|---|---|
| `time_per_unit` | 3.5h/건 | 낭비 계량 |
| `freq_per_month` | 20건/월 | 낭비 계량 |
| `cost_per_month` | 140만원 | 우선순위 공식 (`아픔 = 시간×시급 + 실수손실×빈도`) |
| `pattern_id` | P07 | O2 (처방 매핑) · O4 (승격 판정) |
| `lane` | `time`(시간 해방—환산 가치) \| `cash`(현금 누수—실질 절감) | O1·O2 — **보고서는 두 레인을 절대 섞지 않는다.** 판별: 규칙으로 환원되는가(판단 불필요) + 실제 현금이 나가는가. 인건비 환산은 무조건 `time` |

> 인터뷰 4단계("아픔 계량")가 정확히 이 필드를 채우는 단계다. 계량 없는 병목은 `status: 미확인`으로 저장하고 리포트에서 구분한다.

### edges

| 필드 | 제약 |
|---|---|
| `tenant` / `source_id` / `target_id` | 필수 |
| `verb` | **12개 어휘만** (만든다·승인한다·확인한다·전달한다·적는다·참조한다·기다린다·판단한다·소유한다·요청한다·알린다·대사한다). 위반 시 저장 거부 — 어휘가 흔들리면 O4(교차 테넌트 비교)가 무너진다 |

### patterns (카탈로그 — 테넌트 무관, 잡스랩 자산)

| 필드 | 내용 |
|---|---|
| `id` | P01~P12 (키트 v0.1 시드) |
| `name` / `signal` / `cause` | 패턴명·현장에서 들리는 말·전형 원인 |
| `remedy` | 2주 처방 템플릿 |
| `observed_tenants` | 관찰된 테넌트 목록 — **2개 이상이면 승격 후보** (O4는 이 필드 하나로 계산된다) |

### snapshots (O5 전용)

| 필드 | 내용 |
|---|---|
| `tenant` / `taken_at` | 시점 |
| `bottleneck_metrics` | 그 시점의 병목별 계량값 사본 |

개선 작업 전과 후에 한 번씩 찍는다. "월 70시간 → 15시간"이라는 문장은 스냅샷 두 장의 뺄셈이고, 이게 공개 사례와 구독 갱신 설득의 재료다.

## 5. 산출 파이프라인 (input이 끝이 아니라 시작)

```
인터뷰(guide) ──→ nodes(status: 미확인) ──확정──→ nodes(확인됨)
                                                    │
              ┌─────────────┬──────────────┬────────┴──────┐
              ▼             ▼              ▼               ▼
        O1 병목 리포트  O2 개선 제안서  O3 컨텍스트 팩   O4 패턴 승격
        (질의 + 템플릿) (patterns 조인)  (에이전트용 덤프)  (observed_tenants ≥ 2)
                                                    │
                                        개선 실행 후 ▼
                                              O5 전후 비교 (snapshots)
```

각 산출물은 "DB 질의 + 고정 템플릿"으로 생성한다 — 사람이 보고서를 다시 쓰는 순간 이 시스템의 존재 이유가 사라진다.

## 6. 구현 우선순위 (최소 구현)

1. **nodes + edges + 어휘 검증** — 인터뷰 산출이 처음부터 규격대로 쌓이게 (지금 안 하면 나중에 정제 비용)
2. **O3 컨텍스트 팩** — 가장 싸고 가장 빨리 돈과 연결됨 (에이전트에게 테넌트 덤프 한 방)
3. **O1 병목 리포트** — A 상품 산출물 ② 자동 생성
4. patterns 테이블 + O2/O4 — 2호 고객부터 의미 생김
5. snapshots + O5 — 첫 개선 작업 직전에 1장 찍는 것부터

## 7. 하지 말 것

- 질문 목록에 없는 필드 추가 ("나중에 쓸지도 몰라" — 안 쓴다)
- 12개 어휘 밖의 관계 저장 허용 (교차 비교 자산이 죽는다)
- evidence 없는 병목을 확인됨으로 저장 (리포트 신뢰 붕괴)
- 산출물을 손으로 작성 (질의+템플릿으로만)
